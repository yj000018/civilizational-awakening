# Translation System — Static DeepL Pipeline

Reusable static translation module for React + Vite + Markdown sites.
Built on Civilizational Awakening. Copy the module files to any site.

---

## Architecture

```
English source (content/en/)
  → git push to main
  → GitHub Action detects changed .md files
  → scripts/translate-site.ts
  → DeepL API (server-side only, key never exposed to client)
  → content/fr/ + content/it/ (generated, committed)
  → Vite build reads all locales
  → Static site deployed
  → User selects locale via GlobeLanguageSwitcher
  → React reads pre-translated content from correct locale dir
```

**Key principle:** No runtime translation for visitors. All translations are pre-generated static files.

## File Structure

```
scripts/
  translate-site.ts         ← Main translation script (reusable)

.github/workflows/
  translate.yml             ← GitHub Action (auto-translate on push)

client/src/
  content/
    en/                     ← Source content (English, human-authored)
      pillars/
      projects/
      concepts/
      thinkers/
      essays/
    fr/                     ← Auto-generated by DeepL (do not edit manually)
    it/                     ← Auto-generated by DeepL (do not edit manually)

  lib/
    content.ts              ← Content loader (reads from en/ by default)
    i18n.ts                 ← Locale state, SUPPORTED_LOCALES, helpers

  contexts/
    LocaleContext.tsx       ← React context: locale + setLocale

  components/
    GlobeLanguageSwitcher.tsx  ← Globe icon in nav, EN/FR/IT dropdown
    LanguageSwitcher.tsx       ← Google Translate fallback (disabled by default)
```

## Environment Variables

| Variable | Required | Default | Description |
|---|---|---|---|
| `DEEPL_API_KEY` | Yes (script only) | — | DeepL Free or Pro key. Never exposed to client. |
| `TRANSLATION_SOURCE_LOCALE` | No | `en` | Source language code |
| `TRANSLATION_TARGET_LOCALES` | No | `fr,it` | Comma-separated target locales |
| `VITE_ENABLE_GOOGLE_TRANSLATE` | No | `false` | Re-enable GT fallback in footer |

### DeepL Free vs Pro

The script auto-detects based on the key suffix:
- Free keys end with `:fx` → uses `api-free.deepl.com`
- Pro keys → uses `api.deepl.com`

Free tier: **500,000 characters/month** (~83,000 words). Sufficient for incremental updates on a ~25k word site.

## Running the Translation Script

```bash
# Translate all modified files (incremental, hash-based)
DEEPL_API_KEY=xxx tsx scripts/translate-site.ts

# Force retranslate all files (ignore hash cache)
DEEPL_API_KEY=xxx tsx scripts/translate-site.ts --force

# Translate a single file
DEEPL_API_KEY=xxx tsx scripts/translate-site.ts --file client/src/content/en/pillars/mind.md
```

## How Hash-Based Skip Works

Each translated file gets a `translation` block in its frontmatter:

```yaml
translation:
  source_locale: en
  target_locale: fr
  source_file: client/src/content/en/pillars/mind.md
  source_hash: a3f8b2c1d4e5f6a7
  provider: deepl
  translated_at: 2026-06-06T09:00:00.000Z
```

On next run:
1. Script reads `source_hash` from existing translated file
2. Computes SHA-256 of current source file
3. If hashes match → **skip** (no API call, no cost)
4. If hashes differ → **retranslate**

To force retranslation of a specific file: delete the translated file, or use `--force`.

## Adding a New Language

1. Add the locale to `TRANSLATION_TARGET_LOCALES` in `.env` or the GitHub Action
2. Add the DeepL locale code to `CONFIG.deeplLocaleMap` in `scripts/translate-site.ts`
3. Add the locale to `SUPPORTED_LOCALES` in `client/src/lib/i18n.ts`
4. Add the locale to `LOCALE_FILES` in `client/src/lib/content.ts` (add a new `import.meta.glob`)
5. Run the script: `DEEPL_API_KEY=xxx TRANSLATION_TARGET_LOCALES=fr,it,de tsx scripts/translate-site.ts`

## What Is NOT Translated

The script protects these elements from DeepL:

| Element | Method |
|---|---|
| Fenced code blocks ` ``` ``` ` | Replaced with placeholder before API call |
| Inline code `` `code` `` | Replaced with placeholder |
| URLs `https://...` | Replaced with placeholder |
| Template variables `{{ }}` | Replaced with placeholder |
| JSX/HTML components `<Component />` | Replaced with placeholder |
| Markdown images `![alt](url)` | Replaced with placeholder |
| Frontmatter technical fields | `type`, `slug`, `status`, `tags`, etc. — skipped entirely |
| Frontmatter translated fields | `title`, `summary`, `one_liner`, `core_question` — translated |

## GitHub Action

File: `.github/workflows/translate.yml`

**Trigger:** Push to `main` when `client/src/content/en/**/*.md` files change.

**Loop prevention:** The auto-commit uses `[skip ci]` in the commit message, which prevents the action from re-triggering itself.

**Manual trigger:** Go to Actions → "Auto-translate content (DeepL)" → Run workflow. Optional `force` checkbox to retranslate all files.

### Required GitHub Secret

Add in: Repository → Settings → Secrets and variables → Actions → New repository secret

```
Name:  DEEPL_API_KEY
Value: your-deepl-api-key-here
```

## Reactivating Google Translate Fallback

The GT fallback is preserved in `client/src/components/LanguageSwitcher.tsx` but disabled by default.

To re-enable:

1. Add to `.env`:
   ```
   VITE_ENABLE_GOOGLE_TRANSLATE=true
   ```

2. In `Layout.tsx`, uncomment the `<LanguageSwitcher />` import and render inside the `{ENABLE_GOOGLE_TRANSLATE && ...}` block.

3. The GT CSS suppression (hiding Google's injected toolbar/banner) remains active regardless of the flag — it prevents any GT chrome from leaking if GT is partially loaded.

## Reusing This Module on Another Site

Copy these files:

```
scripts/translate-site.ts           ← adjust CONFIG.contentRoot
.github/workflows/translate.yml     ← adjust paths if content structure differs
client/src/lib/i18n.ts              ← adjust SUPPORTED_LOCALES if needed
client/src/contexts/LocaleContext.tsx  ← no changes needed
client/src/components/GlobeLanguageSwitcher.tsx  ← no changes needed
client/src/components/LanguageSwitcher.tsx       ← GT fallback, keep as-is
```

Then:
1. Update `content.ts` to add `import.meta.glob` for each locale
2. Update `pathToTypeAndSlug()` if your content structure differs
3. Add `DEEPL_API_KEY` to GitHub Secrets
4. Push a `.md` change to `content/en/` to trigger first translation

## Known Limitations

- **DeepL Free tier:** 500k chars/month. Sufficient for incremental updates. Monitor at [deepl.com/account/usage](https://www.deepl.com/account/usage).
- **Locale switch = content reload:** The current implementation re-reads content from the correct locale directory on locale change. No page navigation required, but content re-renders.
- **No URL-based locale routing:** URLs stay the same (`/pillars/mind`) regardless of locale. If SEO-critical locale URLs are needed (e.g., `/fr/pillars/mind`), a more significant routing refactor is required.
- **Translated files in git:** `content/fr/` and `content/it/` are committed to the repo. This is intentional — it makes the build reproducible without needing DeepL at build time.
